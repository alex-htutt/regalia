# Dashboard — Context

> Dense, manually maintained snapshot of the `dashboard/` project. Prime any chat with `@file dashboard/_context_dashboard.md`.
> Build/run details and agent rules live in [[dashboard/CLAUDE.md|CLAUDE.md]]; the vault-root `CLAUDE.md` holds the full v1.x feature log.

## Overall goal

Grow the Flask `dashboard/` from a read-only vault viewer into a **self-hosted, agentic workspace** — a personal cockpit that reads the vault, chats across local + cloud models, and runs agents against task notes. Borrow *architecture shapes* from **Odysseus** (https://github.com/pewdiepie-archdaemon/odysseus), not its code (AGPL, different stack). Single-user, personal tool; may ship to GitHub later, no pressure.

**Guiding decisions:**
- **Build solo (Path C):** keep extending Flask + vanilla JS. Do not fork or run Odysseus alongside.
- **Hybrid models (three tiers):** local Ollama = `fast` (cheap/private); Claude Code CLI = `claude` (subscription-billed, powers Chat + Twin); Anthropic API = `smart` (API-credit billed, powers the Agents view's tool use). See v1.6 below.
- **Vault is the database:** markdown + YAML frontmatter is the source of truth. Defer a vector DB until retrieval actually hurts. Skip Docker / auth / multi-user.

## Architecture (where things live)

- `app.py` — vault walk, frontmatter parsing, all routes (`/api/tasks`, `/browse`, `/recent-folders`, `/usage`, `/news`, `/project`, `/agents`, `/agent/run`, `/agent/run/<id>`, `/chat`, `/chat/upload`, `/router/status`, `/twin/chat`, and the v1.19 email routes `/inboxes`, `/inbox/<id>`, `/email/<id>/<msg_id>`, `/email/draft`). Project scaffolding extracted into `create_project_core()` (shared by the route and the agent tool). Holds the in-memory agent-run store + background-thread runner, `_vault_outline()`/`_vault_chat_system()` (v1.7, fast-tier chat grounding), and `_rel_time()`/`api_recent_folders` (v1.14, landing parallax). Email routes are thin wrappers over `mailbox.py`.
- `static/ascii-dither-background.js` — self-contained third-party ASCII-dither + WebGL hero background (copied from `Assets/ascii_js.js`); auto-mounts into `[data-ascii-dither-bg]`. Only third-party JS; no build, no deps.
- `router.py` — the model-routing primitive. Every model call goes through `chat()` / `chat_tools()`; new backends = one more branch here. Three tiers: `fast` (Ollama/urllib), `smart` (Anthropic SDK), `claude` (Claude Code CLI subprocess, subscription-billed). `chat_tools()` is `smart`/`fast` only — the CLI tier is plain-chat (no structured tool use).
- `agent.py` — the agent loop: vault tools + **email tools** (`list_inboxes`/`read_inbox`/`search_email`/`read_email`/`draft_email`, all delegating to `mailbox.py`), the `AGENTS` registry (incl. **Inbox Triage**), `chat_vault()` (read-only/edit chat loop), `run_agent()`, and `_run_agent_claude()` (the streamed `claude`-tier runner). Imports `app` **lazily** (inside tool fns) to dodge a circular import; imports `mailbox`/`router` at top (neither imports `app`).
- `mailbox.py` — the email engine (v1.19): per-account token store (`.email_tokens/`, gitignored), token refresh (google-auth / msal, lazy-imported), and a provider-agnostic API over Gmail REST + Microsoft Graph via stdlib `urllib`. **Drafts-only — no send function exists.** Tokens never logged/returned; account ids traversal-guarded.
- `email_sources.py` — email config only (provider endpoints, OAuth scopes, fetch limits, cache TTL; OAuth *client* creds read from env). Mirrors `news_sources.py`.
- `connect_email.py` — one-time interactive OAuth consent CLI (`python connect_email.py gmail|outlook`); deliberately not a Flask route (the consent flow blocks on a browser).
- `templates/index.html` — single-page UI (inline CSS + vanilla JS), no build step. Agents view streams run steps via polling; **Inbox** view (account switcher + list + reading pane + Save-draft modal).
- `desktop.py` — pywebview native-window shell over the same Flask app.

## Active deliverable(s) / next steps

- **Agent-runtime switch (2026-06-27 decision; finalized after step-0 spike — no-new-deps gate is dropped, see [[dashboard/CLAUDE.md|CLAUDE.md]]):** route **both** Inbox Triage *and* Research onto **Hermes** (Nous Research, github.com/NousResearch/hermes-agent, MIT). **OpenClaw was dropped** — it's a Node runtime that autonomously sends mail, whereas Hermes is Python (matches the stack), has Gmail/Outlook email skills *and* web-research tools, and runs on one Portal OAuth login. **Spike correction:** Hermes is *not* a Python library — it's a full agent CLI with its own tool loop, so the integration is the **`claude`-tier pattern** (shell out to the CLI, map its stdout onto our step vocab via a runner like `_run_agent_claude`), **not** an adapter behind `mailbox.py`/`news_sources.py`. Those stdlib modules stay as-is (drafts-only fallback + the existing Inbox UI).
  - **Auth: account tokens, not API cost — confirmed.** Hermes runs on the **Nous Portal** via `hermes setup --portal` + `--provider nous` (subscription credits, no per-request API keys).
  - **Headless seam (confirmed step 0):** `hermes -z "<task>"` prints only the final reply to stdout (clean subprocess capture); per-run override via `--provider`/`--model` or `HERMES_INFERENCE_MODEL`.
  - **Decisions (2026-06-27):** (1) **Send-after-review** replaces structural drafts-only — the agent drafts, the human reviews & clicks **Send** in the Inbox view (human-in-the-loop is the gate). Add a cheap *deterministic* pre-send lint (recipients present, no leftover placeholder/`[TODO]` text, warn on reply-all or empty subject) — **not** a second LLM reviewer agent (reserve that only for future *unattended* cron sends, which needs a new send path in `mailbox.py`). (2) **Tool bypass accepted** — Hermes uses its own tools, bypassing our in-process vault/email tools (same tradeoff as the `claude` tier).
  - **Inbox access (confirmed):** Hermes reads the inbox via its **own** account connection (separate from the dashboard's `.email_tokens/`, so a one-time reconnect): Google Workspace skill (Gmail OAuth2, `$GAPI gmail search/get`) or Himalaya IMAP/SMTP (Gmail/**Outlook**/Yahoo/Fastmail).
  - **Inbox retrieval → Hermes (2026-06-27):** **all** mail retrieval — for both the triage agent *and* the human-facing Inbox view — moves to **Hermes's email skills** (Google Workspace Gmail OAuth2, or Himalaya IMAP/SMTP for Outlook/others). **`mailbox.py` keeps its UI role** (the `/api/inboxes`, `/api/inbox/<id>`, `/api/email/...` routes + the message dict shapes the Inbox view renders), but its **provider internals are swapped**: the `_gmail_*` / `_graph_*` / `_api` / token layer is replaced by a Hermes-backed retrieval. *Why (user's reason):* **avoid the Google API + Azure dynamics** — no more Google Cloud "Desktop app" client secret, no Azure app registration, no `GMAIL_OAUTH_CLIENT` / `MS_OAUTH_CLIENT_ID` / `MS_OAUTH_TENANT`, and `google-auth-oauthlib` + `msal` drop out. Hermes owns all provider auth via **one** `hermes setup` email connection, which serves both the UI and triage.
  - **Stays in the dashboard:** the Inbox view UI, the human **Review & Send** gate, and the deterministic pre-send lint. Drafting/sending also routes through Hermes (Hermes holds thread context + the original message id), surfaced in the dashboard for review.
  - **Key step-1 risk:** turning Hermes's agent output into the **structured** message shapes the UI needs (id / from / subject / date / unread). Hermes has `--json` on some commands and an email skill (`$GAPI gmail search/get`); need to confirm a reliable structured-output path (JSON-mode prompt or a skill command that emits parseable data) rather than scraping free-text — verify first in step 1.
- **Email follow-ups (v1.19 shipped read + drafts):** (a) once an inbox is connected, verify live end-to-end (drafts land in the real Gmail/Outlook Drafts); (b) optional **send-with-confirm** — a deliberate trust-model change (would add `Mail.Send` + a Gmail send call + a UI confirm); keep drafts-only until the write-confirm gate exists; (c) a scheduled **Inbox Triage** run once Stage 4 lands.
- **Stage 4 — automations (the headline next):** in-process scheduler (e.g. APScheduler — deps are now fair game, no approval gate) running agents on a cron, e.g. a morning standup or inbox triage. Reuses `run_agent()` verbatim. Gated behind the trust floor below.
- **Trust floor (precede unattended cron):** run-history persistence (persist `_RUNS` to JSON/SQLite) + a **write-confirm gate** for `write_note`/`create_project`/`draft_email`.
- **Stage 5 — memory/embeddings:** only if agents start fumbling context. Then add a vector store. Not before.
