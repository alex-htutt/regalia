# Dashboard — Agent Rules

## Commands (run from `dashboard/`)
- **Run (browser):** `python app.py` (or `start.bat`) → `http://localhost:5000` (debug mode, hot-reloads `app.py` + templates).
- **Run (desktop window):** `python desktop.py` (or `desktop.bat`) — native pywebview window, same app.
- **Test:** `python -m unittest discover -s tests` — stdlib `unittest`, no runner dep. **Run before and after any change to `app.py`/`agent.py`/`router.py`** (the trust floor for unattended automation — see `PRODUCT_VISION.md`).
- **Build CSS:** `build-css.bat` (one-off) or `watch-css.bat` (live); manual: `tools\tailwindcss.exe -i tailwind.input.css -o static\tailwind.css --minify`. Only needed when Tailwind utility classes change.
- **Sync ASCII hero:** `sync-ascii.bat` after re-exporting `Assets/ascii_js.js`.
- **Fetch Tailwind binary:** `get-tailwind.bat` (the ~108MB `tools/tailwindcss.exe` is gitignored).

## Context
Local Flask web app that reads vault frontmatter and surfaces notes as a task dashboard.

Two ways to run the **same** app:
- **Browser:** `python app.py` (or `start.bat`) → `http://localhost:5000`.
- **Desktop window:** `python desktop.py` (or `desktop.bat`) → native pywebview window wrapping the Flask app on an OS-assigned port. Same code, same UI; just a different shell. External (http/https) links open in the system browser instead of inside the window.

## Stack
- Backend: Python / Flask (`app.py`)
- Model router: `router.py` — `chat(messages, tier)`, `chat_tools(messages, tools, tier)`, and `claude_code_stream(prompt, …)` over three backends: `fast`=Ollama local, `smart`=Anthropic cloud API (API-credit billed; drives the Agents view's in-process tool loop), `claude`=Claude Code CLI subprocess (bills your Claude subscription, not API credits; used by Chat + Twin, and now the Agents view too — see below). Every model call goes through here. The `claude` paths strip `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN` from the subprocess env so they always use subscription (OAuth) auth, never API billing.
- Agent loop: `agent.py` — vault tools + `AGENTS` registry + `run_agent()`; drives the model→tools→results cycle for the Agents view. Imports `app` lazily (inside tool fns) to avoid a circular import. On the `fast`/`smart` tiers it drives its own in-process loop (`router.chat_tools` + the custom vault tools); on the `claude` tier it hands off to `_run_agent_claude()`, which streams `router.claude_code_stream()` (the CLI runs its *own* loop with its native Read/Grep/Glob/Write/Edit tools) and maps the CLI's events back onto the same step vocabulary so the UI streams live regardless of tier (v1.18).
- Fast-tier chat grounding (v1.7, extended v1.16): for `tier == "fast"`, `/api/chat` first runs `agent.chat_vault()` — a **read-only tool loop** (`search_vault`/`read_note`/`list_notes`/`list_folder` via `router.chat_tools`) so the local model can actually **open and quote notes**, not just see the layout. The live vault outline (`_vault_outline()`) is still injected into that loop's system prompt to target reads. If the local model can't do tool calls (or any fast-tier `RouterError`), it **falls back** to the old outline-only plain chat (`_vault_chat_system()` → `router.chat()`). Attachment turns skip the tool loop (no `chat_tools` attachment path) and use the image-inlining `router.chat()` with the outline. `smart`/`claude` tiers are untouched (they read files themselves). Needs a tool-capable Ollama model for the loop to bite.
- Frontend: a **single** `templates/index.html` — all markup, the hand-written CSS, and vanilla JS in one file, no JS framework. The whole UI (landing hero + parallax, task list, folder gallery, usage panel, chat, twin, agents view) is client-side rendering against the JSON endpoints. CSS is the hand-written `<style>` block plus a compiled Tailwind utility sheet (see Tailwind below). The product name is **Regalia** (v1.14); the on-disk folder is still `Work_Vault` / `VAULT_ROOT` — don't rename the path (it's wired into git, the vault walk, and the Claude memory store).
- Landing page (v1.13, reworked v1.15): the top of `index.html` is a **pinned-scrollytelling** stage *above* the dashboard — scroll down lands in the overview (`#dashboard`). One tall `.scrolly` track; the inner `.scrolly-stage` pins (`position: sticky`) while a single rAF scroll driver (`window.__scrollyUpdate`) animates in place: title "Regalia." shrinks/fades → "What should we work on?" heading + recent-folder tiles assemble (staggered) → release into the dashboard. Bails to a static stack under reduced-motion / ≤640px (CSS authoritative). The hero background is a self-contained third-party effect, `static/ascii-dither-background.js` (**a copy of `Assets/ascii_js.js`** — re-sync with `sync-ascii.bat` after re-export; auto-mounts into `[data-ascii-dither-bg]`), the only third-party JS, no build/deps. An Aceternity **Spotlight** port (amber `feGaussianBlur` ellipse) fades in above the ASCII bg / below the title. The parallax columns are populated from `/api/recent-folders`. Beams/glass: a fixed `.beams` SVG waves layer (`z-index:0`) sits behind the dashboard; overview surfaces are translucent `--glass*` so they blur it through. Hero canvases are pinned to `object-fit:fill` (NOT cover — cover offsets the script's linear mouse→grid mapping and clips the glow on resize).
- Desktop shell: `desktop.py` (pywebview) — runs Flask in a daemon thread, opens a native window. No effect on the browser path.
- Data source: walks `VAULT_ROOT` (parent of this folder), parses YAML frontmatter
- API: `/api/tasks`, `/api/browse`, `/api/recent-folders` (recently-worked folders for the landing parallax), `/api/usage`, `/api/news` (daily briefing; `?refresh=1` forces refetch), `/api/chat`, `/api/chat/upload`, `/api/chats` + `/api/chats/<cid>` (conversation store CRUD — see Multi-chat below), `/api/router/status`, `/api/ollama/models`, `/api/project`, `/api/twin/chat`, `/api/agents`, `/api/agent/run` (POST → run_id), `/api/agent/run/<id>` (poll)
- Briefing (v1.8, social→mail v1.23): `/api/news` aggregates tech-news RSS + Greenhouse/Lever job postings, all stdlib-only (`urllib`+`xml.etree`+`json`), held in an in-process TTL cache (`NEWS_TTL`, default 30 min). Sources live in `news_sources.py`. Per-source failures degrade gracefully (collected in `errors`), never 500. The panel's middle column is **Important mail** (v1.23): `GET /api/mail/important` → `mailbox.important_messages()` — last-2-days mail from every connected inbox, scored by a deterministic work/school keyword/domain heuristic (config in `email_sources.py`, no model call); fetched separately from the briefing so neither blocks the other.
- Multi-chat (v1.20): the Chat view holds **many concurrent conversations** — a sidebar (new/switch/delete), server-side persistence, and parallel generation. `chats.py` is the store (one JSON file per conversation in the gitignored `dashboard/.chats/`; ids `c_<uuid4hex>`, regex-validated/traversal-guarded); `/api/chats` CRUD persists transcripts. **`/api/chat` generation stays stateless** — concurrency is client-side: per-conversation state objects (`chats{}`/`activeChat` in `index.html`), each send closure captures its conversation id and only touches the DOM when that conversation is visible (else a "new reply" dot on its sidebar entry). Tier/model/Edit-mode are per-conversation and persisted. Transcripts store attachment *metadata* only — the 24h `.chat_attachments/` prune is unaffected. Twin is still single-conversation.
- Email / Inbox (v1.19): connect multiple inboxes (Gmail + Outlook), read them, and save **drafts** (never send). `email_sources.py` = config only (provider endpoints, OAuth scopes, limits — mirrors `news_sources.py`); `mailbox.py` = the engine (token store, read, drafts) with a provider-agnostic API; `connect_email.py` = one-time interactive OAuth consent CLI. Endpoints: `GET /api/inboxes`, `GET /api/inbox/<account_id>?q=&limit=`, `GET /api/email/<account_id>/<msg_id>`, `POST /api/email/draft`. Agent tools (`list_inboxes`/`read_inbox`/`search_email`/`read_email`/`draft_email`) + an **Inbox Triage** agent. See the "Email" section below before touching any of it.

## Behavior in this folder

**Read `app.py` first** before suggesting any change — the vault walk, ignore lists, and frontmatter parsing are tightly coupled. Breaking the walk silently breaks everything.

**Plan in two parts — macro + micro** (vault-wide rule, see root `CLAUDE.md`). Before building any new feature here, split the plan: the **macro plan** is the conventional high-level view (goal, horizon, scope/scale, milestones, constraints — *what* and *why*); the **micro plan** is the concrete build blueprint (which modules/files/functions change, data shapes, the interfaces between parts, new endpoints, deps, and build order — *how*). Given this app lives in a few tightly-coupled files (`app.py`, `agent.py`, `router.py`, the one `index.html`), the micro plan should name the exact touch points and how they wire together so the build doesn't improvise structure. If problems arise mid-build, revise the **micro plan** (re-split components, change interfaces) and keep the macro plan stable unless the goal itself changes.

**Dependencies are welcome — no approval gate.** Add a third-party library when it earns its place; reach for stdlib for trivial things, but don't contort the code to avoid a dep. Soft guideline only: prefer **mature, widely-used, well-maintained** packages, keep the footprint sane, and pin the addition in `requirements.txt` with a one-line note on what it's for. The current baseline is Flask, PyYAML, anthropic, pywebview, plus stdlib; `pywebview` is only needed for the desktop window — the browser path (`app.py`) still runs on Flask + PyYAML + stdlib alone. The Tailwind toolchain (below) adds **no Python dep** — it's a single standalone binary, not a `pip`/`npm` package.

**Frontend changes:** Markup and the hand-written CSS/JS live in the one `templates/index.html`, served directly. There is now **one build artifact**: `static/tailwind.css` (compiled from `tailwind.input.css`). If you add/change Tailwind utility classes in the markup, rebuild it (see Tailwind). Editing only the hand-written `<style>` block or the JS needs no rebuild. Test in-browser after any change.

**ASCII hero asset:** `static/ascii-dither-background.js` is a **copy** of `Assets/ascii_js.js` (the served artifact) — it does **not** auto-update. After re-exporting the ASCII art, run **`sync-ascii.bat`** (copies Assets → static) and hard-refresh. As long as the export keeps the `data-ascii-dither-bg` mount, no HTML/CSS changes are needed.

## Tailwind (compiled, standalone CLI)

Tailwind v4 is compiled by the **standalone CLI binary** (`tools/tailwindcss.exe`, ~108MB) — no Node, no `npm`, no `node_modules`. The binary is **gitignored**; re-fetch it with `get-tailwind.bat`.

- **Input:** `tailwind.input.css` — imports the theme + utilities layers, our Hyperstudio design tokens as a v4 `@theme` (so `bg-obsidian-canvas`, `text-frost-text`, `text-amber-whisper`, `rounded-cards`, `font-aeonik`, … exist), and `@source "./templates/**/*.html"` for purge.
- **Output:** `static/tailwind.css` (committed, ~5KB minified; Flask serves it; linked in `index.html` *before* the hand-written `<style>`).
- **Build:** `build-css.bat` (one-off, minified) or `watch-css.bat` (live rebuild while editing markup). Manual: `tools\tailwindcss.exe -i tailwind.input.css -o static\tailwind.css --minify`.
- **Preflight is intentionally NOT imported** — Tailwind's base reset would break the hand-written UI (e.g. markdown list bullets). We pull in only `theme` + `utilities`.
- **Cascade:** utilities live in `@layer utilities`; the page's existing `<style>` is **unlayered**, so unlayered rules always win. This means Tailwind cannot regress the existing look, and utilities apply cleanly to **new/pasted** markup. To override an existing styled element with a utility, use the important modifier (`bg-red-500!`).
- **Workflow gotcha:** purge is content-driven — a utility class only lands in `static/tailwind.css` if it appears in a scanned template (or the `@source inline(...)` safelist in `tailwind.input.css`). Add classes → rebuild → hard-refresh. A class that "does nothing" usually means you forgot to rebuild.
- **Template-literal gotcha (bit us once):** the UI is rendered with JS template strings, so a class glued directly to a `${...}` interpolation is invisible to Tailwind's scanner — `leading-[1]${warn?...}` won't compile the `leading-[1]`. Always leave a space before the interpolation: `... leading-[1] ${warn ? 'text-overdue' : ''}`. Conditional classes inside the `${}` must be plain, space-bounded literals so they're statically detectable.
- **Converted so far:** the overview stat cards (`#stats` container + the cards built in `render()`) are pure Tailwind utilities (the old `.stat`/`.stats` CSS was deleted) — use them as the reference pattern for converting other panels. Everything else is still hand-written CSS.

**Tier gotcha — three tiers, two agent code paths (v1.18):** `chat()` and `/api/agent/run` both now accept `fast`/`smart`/`claude`. But the Agents view runs them via **two different mechanisms**: `fast`/`smart` drive the **in-process tool loop** (`run_agent()` → `chat_tools()` → the custom vault tools — needs native tool use, which only Ollama/Anthropic expose), while `claude` hands off to **`_run_agent_claude()`** which streams the CLI's *own* agentic loop (`router.claude_code_stream`, `--output-format stream-json`) using the CLI's native file tools (Read/Grep/Glob, +Write/Edit for agents that write), **not** the custom vault tools. So the custom vault-tool prompts get a translation note (`_CLAUDE_AGENT_RULES`) on the claude path. Bash is never granted; file access stays confined to `CLAUDE_CLI_CWD`. `--max-turns` does **not** exist in the installed CLI — the run is bounded by the wall-clock `CLAUDE_CLI_TIMEOUT` watchdog instead. Default tiers per surface: Chat → `fast` (cloud button switches to `claude`), Twin → `claude`, Agents → each agent's own default (`fast`/`smart` per the registry), now overridable per-card via the tier `<select>`.

**Circular import:** `app.py` imports `agent` at top; `agent.py` imports `app` **lazily inside tool functions** to break the cycle. Keep agent's module-level imports to stdlib + `router`.

**Privacy:** `app.py` reads token usage from `~/.claude/projects/**/*.jsonl` but intentionally skips message content. Don't add any code that reads or logs message content.

**Email (v1.19) — drafts-only, by construction.** The Inbox view and email tools can read mail and create **drafts**, but **nothing in the codebase can send**. This is enforced three ways, all load-bearing — do not "add a send button" without an explicit decision to change the trust model: (1) `mailbox.py` has **no** send function; (2) the Microsoft Graph token is requested **without `Mail.Send`** (`email_sources.MS_SCOPES`), so the Outlook token literally cannot send; (3) the Gmail code never calls the send endpoint. Smoke tests assert all three. **Auth (#2.5):** OAuth/token handling uses *auth-only* libs (`google-auth-oauthlib` for Gmail, `msal` for Outlook), imported **lazily** so the rest of the app runs without them installed (they're only needed once you connect an inbox); the actual mail I/O is plain `urllib`+`json` against the Gmail REST API / Microsoft Graph (no heavy SDKs). **Secrets:** OAuth *client* creds come from env (`GMAIL_OAUTH_CLIENT`, `MS_OAUTH_CLIENT_ID`, `MS_OAUTH_TENANT`); per-account tokens live in the gitignored `dashboard/.email_tokens/` store. **Never log or return tokens** (a smoke test guards this) — `list_accounts`/`accounts_overview` expose only id/provider/address/unread/status. Account ids are traversal-guarded into the token store (`mailbox._account_path`). Connecting an inbox is a one-time CLI step (`python connect_email.py gmail|outlook`), deliberately *not* a Flask route (the consent flow blocks on a browser). **Tier note (v1.22):** the custom in-process email tools (`list_inboxes`/`read_inbox`/…) run on `fast`/`smart` via native tool use; on the **`claude`** tier they ride over **MCP** instead — `mail_mcp.py` is a stdio MCP server (official `mcp` SDK, imported only by that subprocess) wrapping the same `agent._tool_*` email functions, attached per-run by `agent._mailbox_mcp_config()` + `router.claude_code_stream(mcp_config=…)` with only `mcp__mailbox__<tool>` granted (no Bash, no filesystem tools for email-only agents). **Inbox Triage therefore defaults to `claude`** — subscription-billed, no API key. There is still no send tool on any path. *(History: a Hermes agent-runtime switch was built partway through v1.19–v1.20 and then **reverted** on 2026-07-09 to keep dependencies minimal — the inbox manager is this home-built OAuth engine; see `VERSION_HISTORY.md`.)*

**Dev server:** `python app.py` — Flask runs in debug mode. Changes to `app.py` hot-reload automatically; template changes do too.

**Tests:** `python -m unittest discover -s tests` (from this folder). Stdlib `unittest` only — no test-runner dependency. The smoke suite (`tests/test_smoke.py`) covers read-endpoint 200s + JSON shape and agent-tool / browser path-traversal safety. Run it before and after any change to `app.py`, `agent.py`, or `router.py`; it's the trust floor for the Stage-4 unattended automation (see `PRODUCT_VISION.md`).
